# XRK-AGT 底层架构设计

> 目标：作为项目“底层设计与边界”的单一事实源，统一 Runtime / Infrastructure / Core 业务层、AI 调用链路、配置与扩展约束。  
> 适用范围：`src/`、`core/*/`、`config/`、`data/server_bots/{port}/`、`subserver/pyserver/`。  
> 对齐文档：`PROJECT_OVERVIEW.md`、`docs/README.md`、`docs/aistream.md`、`docs/factory.md`、`docs/subserver-api.md`。

---

## 1. 架构分层与职责

### 1.1 运行核心层（Runtime）

- 路径：`src/bot.js`
- 职责：
  - 启动 HTTP/HTTPS/WebSocket 服务
  - 管理全局对象 `Bot`、事件总线、生命周期
  - 挂载中间件、鉴权、路由入口

### 1.2 基础设施层（Infrastructure）

- 路径：`src/infrastructure/`、`src/utils/`、`src/factory/`
- 职责：
  - 提供 Loader、Base Class、工厂、通用工具
  - 保持抽象稳定，避免承载业务规则
- 约束：
  - 业务实现不得放在 `src/`
  - Core 优先复用基础设施能力，不重复造轮子

### 1.3 业务层（Core）

- 路径：`core/<core-name>/`
- 子目录规范：
  - `plugin/`、`http/`、`stream/`、`tasker/`、`events/`、`commonconfig/`、`www/<app>/`
- 约束：
  - 业务逻辑只放在 Core
  - `www/` 必须有子目录，避免与根 `www/` 冲突

## 2. AI 底层调用架构（权威链路）

### 2.1 主链路

1. 入口：`core/system-Core/http/ai.js`
2. Provider 解析：`LLMFactory.resolveProvider(...)`
3. 客户端创建：`LLMFactory.createClient(...)`
4. 调用模式：
   - 非流式：`client.chat(messages, overrides)`
   - 流式：`client.chatStream(messages, onDelta, overrides)`
5. 工具调用：MCP + tool calling，由 provider 客户端与 MCP 适配层执行

### 2.2 AIStream 链路

1. 入口：`AIStream.process(...)` / `AIStream.execute(...)`
2. 上下文构建：`buildChatContext(...)` + `buildEnhancedContext(...)`
3. LLM 调用：`LLMFactory.createClient(...)`
4. 工具调用：MCP + tool calling
5. 上下文增强：由工作流及其已加载能力执行

### 2.3 子服务端职责边界

- 路径：`subserver/pyserver/`
- 负责：
  - 底层系统接口（如 `/health`、`/api/system/*`）
  - Python 扩展 API 自动装载（`apis/<group>/*.py`）
- 不负责：
  - 直接管理 Node 侧 provider 工厂配置
  - 取代主服务 v3 的统一模型入口

## 3. 配置分层与优先级

### 3.1 关键配置文件

- 默认模板：`config/default_config/*.yaml`
- 运行时配置：`data/server_bots/{port}/*.yaml`
- AI 总配置：`data/server_bots/{port}/aistream.yaml`
- Provider 配置：
  - 官方：`{provider}_llm.yaml`
  - 兼容聚合：`*_compat_llm.yaml`（`providers[].key`）

### 3.2 优先级（LLM 调用）

`请求显式参数 > 运行时 provider 配置 > 工作流默认配置 > 系统默认值`

### 3.3 重要约定

- `/api/v3/chat/completions` 的 `model` 对外语义是 **provider key**。
- 真实模型名由 provider 配置内 `model/chatModel` 决定。
- 工作流扫描固定走 `core/*/stream/*.js`。

## 4. 扩展策略

### 4.1 新增业务能力

- 插件：`core/*/plugin/*.js`
- API：`core/*/http/*.js`
- 工作流：`core/*/stream/*.js`

### 4.2 新增 LLM Provider

1. 在 `src/factory/llm/` 增加客户端实现，或在 compat 配置新增 `providers[].key`
2. 对齐 `core/system-Core/commonconfig/*.js` schema（如需前端可配置）
3. 补齐默认配置模板与运行时配置说明

### 4.3 文档更新原则

- 代码优先：文档与代码冲突时，以代码行为为准
- 一次改动，至少同步更新：
  - 本文档
  - 对应子系统文档
  - 导航文档 `docs/README.md`

---

## 5. 非功能性设计要求

- 稳定性：优先可回退、可观测、可降级
- 一致性：命名、字段、路径在代码与文档中保持一致
- 可运维：异常日志可定位到 provider/workflow/route
- 可演进：新增 provider 或 workflow 不影响既有接口
