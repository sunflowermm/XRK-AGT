# 代码审查清单

发布前按序自检；**自动化项以 `pnpm test` / `pnpm test:e2e` 为准**（数据见 [框架测试指南.md](框架测试指南.md)）。

## 1. 自动化（必过）

```bash
pnpm test        # 目标：60 pass（仅 system-Core 基准），~3s
pnpm test:e2e    # 目标：3 pass，Bot 能监听并响应 API
```

## 2. 架构边界

| 检查项 | 期望 |
|--------|------|
| 业务只在 `core/<name>/` | 无业务写入 `src/` |
| HTTP 继承 `HttpApi` | `core/*/http/*.js` |
| 带 `init()` 的 API | 依赖 `initHook.call(this)`（`src/infrastructure/http/http.js`） |
| Tasker 不在模块顶层 `Bot.tasker.push` | 无 `Bot` 时 import 不抛错 |
| 渲染器 | `import` 不启动浏览器；截图前 `ensureLoaded()` |
| 框架测试口径 | 基准数量只指 `core/system-Core/`，扩展 core 不计入 |

## 3. 安全

| 检查项 | 期望 |
|--------|------|
| `/api/*` 默认鉴权 | `HttpApi` + `auth.js` |
| 公开路由 | `systemAuth: false` |
| 文件路径 | `InputValidator` / `assertPathUnderRoots` |
| 生产外网 | 非 127 访问 `/api/*` 需 Key |

## 4. 配置三件套

新增/改字段须同时维护：

1. `config/default_config/<name>.yaml`
2. `core/*/commonconfig/` 或 `system.js` schema
3. 消费代码（`cfg.*` / `getAistreamConfigOptional`）

## 5. system-Core 模块数量（与测试对齐）

| 类型 | 基准 |
|------|------|
| HTTP | 11 |
| stream | 7（不含 screen、react-bits-mcp） |
| plugin | 15（不含远行商人等个人插件） |
| tasker | 4 |
| events | 3 |

扩展 core 增减不影响上表；勿将全仓库加载日志中的合计数（如 HTTP 19）当作框架达标线。

## 6. 启动冒烟（人工，改底层后必做）

```bash
node app
```

- 日志出现 HTTP 监听端口与「启动完成」
- 浏览器打开 `http://127.0.0.1:<port>/xrk`
- 首次截图/渲染功能仍可用（验证渲染器懒加载未破坏业务）

## 7. 底层改动记录（2026-05-17）

| 改动 | 文件 | 影响 |
|------|------|------|
| 渲染器懒加载 | `src/infrastructure/renderer/loader.js` | 启动更快；截图前 `ensureLoaded()` |
| initHook 绑定 this | `src/infrastructure/http/http.js` | 带 `init()` 的 API 可 `wrapHandler` |

| 移除非自带模块 | `screen.js`、`react-bits-mcp.js`、`远行商人.js` | 不计入框架基准；建议放独立 core |
